Chapter 3: Programming Paradigms and Project Organization (Python & R)

Published

Choose the Language for the Task, Not the Tribe

Language tribalism (“I only program in R” or “I only use Python”) is a hallmark of the amateur. R and Python are not religions; they are tools engineered for distinct tasks. Python was built for general-purpose programming, file parsing, and automation. R was built by statisticians for data analysis, modeling, and plotting. A successful researcher uses both, selecting R to test hypotheses and Python to wrangle raw text.

Core Concepts

As a researcher, you do not need to build complex software packages. However, you must write code that is clean, readable, and does not corrupt your data.

To achieve this, you must understand when to deploy Python versus R, how biological files are structured as tables, and how to save your data objects safely.

Python: The Wrangle and Text Processing Tool

Python excels at handling raw files. In genomics, raw data typically arrives as massive text files (FASTQ files of raw reads, FASTA files of genomes, or tabular annotation files like GFF and BED).

Python’s syntax and built-in libraries make it the ideal language for:

  1. Reading files line-by-line: Parsing text files without loading them fully into RAM (as discussed in Chapter 1).

  2. Regular Expressions (Regex): Finding specific sequence motifs, adapters, or primers inside DNA strings.

  3. Command-Line Interfaces (CLIs): Writing scripts that take input file arguments so you can run the same script on multiple samples.

R: The Statistical and Plotting Tool

R is built around the DataFrame—a tabular structure where rows are observations (e.g., genes) and columns are variables (e.g., samples). R is the industry standard for:

  1. Statistical Modeling: Running differential expression (e.g., DESeq2), regression, and multiple-testing corrections.

  2. Visualization: Creating publication-grade plots using ggplot2 (a visual language built on the “Grammar of Graphics” where plots are built in composable layers).

  3. Bioconductor: Accessing a vast repository of packages designed specifically for biological coordinate and genomic data.

Tabular Genomic Formats: BED and GFF as Data Frames

Most genomic annotation files are simply tab-delimited text tables. Understanding how to load these files as data frames is the first step in coordinate math:

  • BED (Browser Extensible Data): A simple table mapping genomic regions. It requires three core columns: chromosome, start position, and end position. Note: BED files use 0-based, half-open coordinate indexing (the start position is 0-indexed, and the end position is not included in the range).

  • GFF (General Feature Format): A more detailed annotation table detailing exons, genes, and transcripts. It uses 1-based, closed coordinate indexing (both start and end coordinates are inclusive and start at 1).

In R, Bioconductor packages translate these text files into validated objects (like GRanges) that handle the coordinate differences automatically, preventing off-by-one errors when computing gene overlaps.

Serialization: R Objects and saveRDS Safety

After running an expensive analysis (like normalizing a single-cell count matrix), you will want to save your R variables to disk so you do not have to rerun the pipeline tomorrow.

In R, there are two primary ways to save files, and one of them is highly dangerous:

The Dangerous Way: save() and .RData

# DO NOT USE THIS FOR SCIENTIFIC DATA
save(my_normalized_counts, file = "my_analysis.RData")

save() exports your variable along with its exact variable name into a binary workspace file. When you load it tomorrow using load("my_analysis.RData"), R restores the variable my_normalized_counts directly into your active workspace. If you already had a different variable named my_normalized_counts in your active session, it will be silently overwritten and erased.

The Safe Way: saveRDS() and readRDS()

# THE CORRECT WAY
saveRDS(my_normalized_counts, file = "my_normalized_counts.rds")

saveRDS() serializes the data structure itself, independent of the variable name. When you load it, you must explicitly assign it to a new variable:

counts_subset <- readRDS("my_normalized_counts.rds")

This preserves your workspace namespaces, prevents accidental variable overwrites, and makes your scripts transparent and readable.

Biological Interpretation

The “Notebook” Pitfall

Many freshman researchers perform their entire analysis in interactive environments like Jupyter Notebooks or RStudio, running blocks of code in random orders. While interactive notebooks are excellent for exploration, they are dangerous for publishing:

  • Out-of-order state: If you run cell 5, then edit cell 2, then run cell 10, your active workspace depends on a sequence of actions that cannot be replicated.

  • Hidden variables: You may delete a line of code, but the variable remains in RAM, letting your script run until you restart the session and it crashes.

The Rule of the Script

Always verify your analysis by restarting your R or Python session and running your scripts sequentially from top to bottom. If the script does not run successfully on a clean workspace, the analysis is not complete.

Summary and Required Reading

  1. Select the tool for the job: Use Python for file preparation and text parsing; use R for statistics and data plotting.

  2. Understand coordinates: BED files are 0-indexed; GFF files are 1-indexed. Use coordinate-aware libraries to handle overlaps.

  3. Use saveRDS: Never use save() to dump workspace images. Use saveRDS() and readRDS() to maintain variable namespace safety.

  4. Enforce sequential execution: Interactive notebooks are for drafts; sequential scripts are for reproducible science.

Required Reading

  • Bioconductor Overview: Familiarize yourself with standard container objects like GRanges and SummarizedExperiment.

  • R ggplot2 tutorial: Learn the layers of the Grammar of Graphics (Data ➔ Aesthetics ➔ Geometries).

Johnson’s Rule: Excel is not a database, and an RWorkspace save file is not an archive. Write clean scripts that read raw tables and output plots. If you cannot replicate your final figures by running a single script on a clean session, your analysis does not exist.
