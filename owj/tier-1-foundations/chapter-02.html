Chapter 2: Organizing and Securing Your Analysis (Git & Conda)

Published

June 5, 2026

Modified

July 5, 2026

An Unrecorded Step is Scientific Fiction

In bioinformatics, your code is your experimental record. If you run a script, generate a figure, and cannot trace the exact code version and software packages that produced it, your results are not science—they are a draft. Bioinformatics reproducibility is not about making your code pretty; it is about ensuring that your future self, your principal investigator (PI), and scientific reviewers can recreate your exact results from your raw data.

Core Concepts

When you start analyzing a biological dataset, the immediate temptation is to write quick, messy scripts and save files all over your desktop. This habit leads to the “reproducibility crisis”: a year later, when a reviewer asks you to re-plot a figure with a new threshold, you will find yourself staring at files named script_v2_final_edit.R or results_table_new_corrected.csv, unable to recall which one is correct.

Reproducibility for a sole researcher requires two simple, non-negotiable tools: project organization (directories), version control (Git), and environment isolation (Conda/Mamba).

Structuring Your Project Directory

Before writing a single line of code, you must establish a standard, consistent folder structure. Do not store your raw sequencing files in the same directory as your scripts or figures.

A standard biological project directory should look like this:

my_project/
├── .gitignore         # Tells Git what files to ignore (e.g., massive fastq files)
├── environment.yml    # Lists all software packages and versions used
├── README.md          # Documents the project overview and execution order
├── data/
│   ├── raw/           # Raw sequencing reads (read-only, never edit)
│   └── processed/     # Intermediate filtered files (e.g., cleaned BAMs)
├── scripts/           # Python/R scripts that execute the analysis
└── results/           # Final figures, tables, and statistics

By keeping raw data strictly read-only and separate from your scripts, you ensure that your raw input remains untainted, and your results are always a predictable output of your scripts.

Layer 1: Version Control with Git

Git is a tool that records changes to your files over time. It creates a cryptographic history of your code. If you make a mistake that breaks your analysis, Git allows you to instantly roll back your files to a working state.

For a sole researcher, the core Git workflow is simple:

  1. Initialize: Create a Git repository in your project folder:

    git init my_project

  2. Inspect: Check which files have changed:

    git status

  3. Stage: Mark changes you want to save:

    git add scripts/normalize_counts.R

  4. Commit: Save the staged snapshot to history with a clear description:

    git commit -m "Add DESeq2 normalization script"

  5. Review: Look at the changes between your current file and the last saved version:

    git diff scripts/normalize_counts.R

What NOT to Track with Git: The .gitignore file

Git is designed for text files (code scripts). It is not designed to track massive binary datasets (such as raw .fastq.gz sequencing reads, 50GB alignment .bam files, or binary R workspaces). If you try to commit large data files to Git, your repository will become bloated, slow, and impossible to upload to repositories like GitHub.

You must create a file named .gitignore at the root of your project folder to tell Git to ignore large data and temporary files:

# Exclude raw and processed data
data/
results/

# Exclude large sequencing formats
*.fastq
*.fastq.gz
*.bam
*.bai
*.vcf
*.sam

# Exclude temporary system files
.DS_Store
.Rhistory
.RData

Layer 2: Environment Isolation with Conda and Mamba

Git tracks the code you write, but it does not track the software packages you run.

If you write an R script that uses the package DESeq2 version 1.38, and a collaborator runs the same script on their laptop with DESeq2 version 1.44, the script may crash or produce slightly different results due to changes in default parameters. This is the package dependency conflict problem.

Conda (and its faster drop-in replacement, Mamba) solves this by creating isolated software environments. Mamba installs tools and libraries inside a dedicated directory, keeping them completely separate from your computer’s default system libraries.

Pinning Your Environment with environment.yml

To make your analysis reproducible, you must document every software package and version you used. You do this by exporting your environment to a configuration file called environment.yml:

name: rna_seq_analysis
channels:
  - bioconda
  - conda-forge
  - defaults
dependencies:
  - python=3.9
  - r-base=4.2.3
  - bioconductor-deseq2=1.38.0
  - bioconductor-salmon=1.9.0
  - samtools=1.17

A collaborator (or your future self) can recreate your exact software environment on a new machine with a single command:

mamba env create -f environment.yml

This guarantees that the exact same package versions are resolved and installed, preventing library conflicts.

Biological Interpretation

For a researcher, reproducibility is not a theoretical exercise; it has direct consequences during peer review.

The Reviewer Mindset: Guarding Against “Silent Shifts”

When you submit a manuscript, a reviewer may request that you rerun your analysis using an updated reference genome or different threshold parameters. If you have not pinned your environment and tracked your code:

  • You will spend weeks trying to reconstruct which versions of software you used.

  • Updating a package globally on your computer to solve one problem will break a different project’s scripts.

  • Your output plots may shift slightly, raising doubts about the robustness of your findings.

A robust directory structure with a version-controlled Git history and a pinned environment.yml means you can address reviewer comments in hours rather than weeks.

Current Landscape

While Conda/Mamba environments are the baseline standard for sole researchers, larger core facilities and clinical labs use containers (like Docker or Singularity) and workflow managers (like Nextflow) to scale analysis across thousands of samples in the cloud. However, as a researcher processing one or two datasets at a time, local Conda/Mamba environments provide the ideal balance of simplicity, speed, and absolute reproducibility.

Summary and Required Reading

  1. Code is your experimental record: Always write scripts; never perform manual Excel formatting on raw data.

  2. Separate data from code: Keep raw data read-only in data/raw/ and script code in scripts/.

  3. Git tracks code, not data: Never commit massive biological files (.fastq, .bam) to Git. Use a .gitignore file.

  4. Mamba isolates environments: Create isolated software environments and pin exact package versions in environment.yml.

  5. Recreate in one step: Use mamba env create -f environment.yml to clone environments across machines.

Required Reading

  • Conda / Mamba Guide: Learn how to create, activate, and export environments.

  • Git Basics: Practice initializing a repository, staging files (git add), and committing changes.

Johnson’s Rule: A peer-reviewed paper whose code cannot be run by a reviewer on a cold machine is a scientific claim without evidence. Pin your software environment, commit your scripts, and treat your analysis as a published protocol.
