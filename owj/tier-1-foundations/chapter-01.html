Chapter 1: The Bioinformatic Computing Environment

Published

June 5, 2026

Modified

July 5, 2026

Compute is a Physical, Not a Magical Resource

All bioinformatics is performed on computers. Every command you run—whether aligning millions of sequencing reads, filtering genomic variants, or plotting gene expression—must direct physical electrons through a silicon chip. If you do not understand the hardware limits of your computer, your analysis will fail. You will try to open a 10-gigabyte sequencing file in a basic text editor or Excel, your system will freeze, and you will not understand why. A professional understands that hardware capacity governs data capacity.

Core Concepts

I suggest you think of your computing environment not as a software sandbox, but as a collection of physical resources. As a biological researcher, you will rarely write complex, multi-node supercomputing scripts. Most of your daily work will involve processing a single dataset at a time on your local laptop, a high-performance desktop workstation, or a remote institutional server. However, biological data is inherently large. Even a single RNA-seq or metagenomic sample can easily exceed 5 to 50 gigabytes of raw data.

To manage this volume without crashing your computer, I will teach you to understand the three physical bottlenecks of computation: CPU (processing power), RAM (working memory), and Storage I/O (disk read/write speed).

The Three Bottlenecks

I recommend you monitor these three resources constantly during any pipeline execution:

  1. CPU (Central Processing Unit): The brain of the computer that executes mathematical instructions. CPU bottlenecks occur when you run computationally intensive algorithms (such as sequence alignment or graph-based assembly). If your CPU utilization is running at 100% across all allocated cores, your task is CPU-bound; it is computation-limited.

  2. RAM (Random Access Memory): The temporary, high-speed workspace of the computer. When a program runs, it loads data into RAM so the CPU can access it instantly. RAM is volatile (erased when the computer turns off) and finite. If your data volume exceeds your physical RAM, the operating system will crash or force-kill the program.

  3. Storage I/O (Input/Output): The read and write channels connecting your long-term storage (SSD or HDD) to the motherboard. Disk I/O bottlenecks occur when the CPU sits idle because it cannot read data off the disk fast enough. If you read a massive file from a slow mechanical hard drive, your pipeline stalls.

The Memory Wall: Why Opening a 10GB File Crashes Your Laptop

A common mistake for young researchers is trying to view a raw biological data file (like a .fastq or .sam file) using default office software (like Microsoft Word, Excel, or TextEdit).

I suggest you understand why this fails: standard applications are designed to load the entire file into RAM before displaying it. If you try to open a 15GB FASTQ file on an 8GB RAM laptop, the operating system is forced to search for space. When physical RAM is exhausted, the OS tries to write temporary data to your hard drive (called “swapping”), slowing your computer to a crawl. Eventually, the Linux kernel invokes a mechanism called the OOM (Out-of-Memory) Killer to force-terminate the text editor to save the machine from freezing.

[!NOTE] The OOM Killer: This is a kernel safety feature. It acts as an automated emergency brake. If you run a script and it suddenly terminates with the word Killed, it almost always means you exceeded your machine’s physical memory.

To process genomic scale datasets, you cannot use office software. I will teach you to use the UNIX command line and the concept of stream processing.

The Leverage of the Terminal: UNIX Streams and Pipes

The UNIX operating system dominates bioinformatics because of a simple, elegant abstraction: everything is treated as a file.

Instead of loading a massive file into memory all at once, the terminal allows you to stream data line-by-line. This is called stream processing—processing data in small, continuous chunks rather than in a single batch. The memory footprint of the program remains constant (usually just a few megabytes), regardless of whether the file is 10 megabytes or 10 terabytes.

The key leverage tool is the pipe operator (|). It redirects the output of one program directly into the input of another, entirely in RAM, without writing intermediate temporary files to your disk.

Let us look at a practical example:

# Novice Approach: writes huge intermediate files to disk, wasting I/O and RAM
grep "chr1" large_dataset.sam > temp_chromosome1.sam
wc -l temp_chromosome1.sam
rm temp_chromosome1.sam

# Professional Approach: streaming data in memory via a pipe
grep "chr1" large_dataset.sam | wc -l

In the professional approach, grep reads large_dataset.sam line-by-line, filters for rows containing "chr1", and pushes only those matching lines directly to wc (word count). The data flows through a small memory buffer in RAM. No intermediate files are written to the disk, saving time and storage space.

Essential Command-Line Streams

I suggest you master a small set of general-purpose UNIX utilities to navigate biological data:

  • cat and zcat: Display files (or compressed .gz files) as a stream.

  • head and tail: View the first or last few lines of a file (essential for checking file formats without loading the whole file).

  • grep: Search for matching patterns (e.g., finding a barcode or primer sequence).

  • awk: A powerful column-oriented text processor (perfect for filtering tab-delimited coordinate files like BED or GFF).

  • sed: A stream editor used to replace text patterns on the fly.

Remote Server Basics: SSH and tmux

As a sole researcher, your laptop will often lack the RAM or CPU cores to run genomic alignments. Your institution will provide access to a remote Linux server.

You connect to this server using SSH (Secure Shell):

ssh username@server.university.edu

Once connected, your terminal is executing commands directly on the remote machine. However, if your internet connection drops—even for a split second—your SSH session will disconnect. The operating system sends a termination signal (SIGHUP) to your running programs, killing your analysis midway.

To prevent this, I recommend you run your commands inside a persistent terminal multiplexer—a utility that maintains active shell sessions independently of your network connection—such as tmux (or screen).

tmux creates an isolated shell session that runs independently of your SSH connection. If your laptop loses power or internet, the process on the server continues to execute uninterrupted.

  • tmux new -s my_analysis: Create a new session named “my_analysis”.

  • Ctrl+b followed by d: Detach from the session (leaving your script running in the background).

  • tmux attach -t my_analysis: Reconnect to your running session from any computer at any time.

Biological Interpretation

When your analysis runs slowly or crashes, I recommend you diagnose the hardware bottleneck before trying to alter your biological parameters or software versions. Let me ask you: if a pipeline runs overnight and fails with a generic error code, how do you distinguish an algorithmic error from an Out-Of-Memory termination? If you cannot answer this, you are wasting compute time.

How to Diagnose Bottlenecks Locally

  1. Check Resource Utilization with htop: I suggest you open a terminal window and run htop to display a real-time visual representation of your CPU cores and RAM usage.

    • If you see all CPU bars pegged at 100%, your task is CPU-bound. If your tool supports multi-threading (e.g., using -t or -p flags in aligners), you can accelerate it by allocating more cores.
    • If you see your memory bar (RAM) full, and the computer is unresponsive, your task is memory-bound. I recommend you rewrite your pipeline to stream the files in smaller chunks rather than loading them in their entirety.

  2. Verify Memory Crashes with dmesg: If your script abruptly stopped and displayed the word Killed, I recommend you run the following command to inspect the kernel buffer:

    dmesg -T | grep -i -E "oom|kill"

    If you see a line stating Out of memory: Kill process, you have hit the memory wall. You must rewrite your code to process data in chunks or migrate to a server with higher memory capacity.

  3. Monitor Disk Capacity with df -h: Before downloading massive FASTQ datasets from the Sequence Read Archive (SRA), I recommend checking your remaining storage:

    df -h

    Running out of disk space midway through a sequence alignment will corrupt your output files, forcing you to restart the process.

Current Landscape

While raw command-line tools are essential, modern bioinformatics has evolved to make remote development more user-friendly:

  • VS Code Remote Development: Instead of editing files on a remote server using complex terminal editors (like vim or nano), you can use Visual Studio Code with the Remote - SSH extension. This allows you to edit files, view plots, and run terminal commands on the remote server using a standard, familiar desktop interface.

  • Mamba/Conda Environment Managers: Software installation has moved away from manual compilation. As we will discuss in the next chapter, package managers allow sole researchers to set up complete environments with a single command.

Summary and Required Reading

  1. Compute is physical: Every command consumes CPU cycles, RAM, and Disk I/O.

  2. Do not load large files into text editors: Use head or zcat to inspect files.

  3. Pipes prevent memory bottlenecks: Use the pipe operator (|) to stream data line-by-line in memory without writing intermediate files to disk.

  4. tmux is your safety net: Run all server analyses inside a tmux session to prevent connection-drop crashes.

  5. Diagnose first: Use htop to identify CPU vs. RAM bottlenecks, and dmesg to check for OOM memory kills.

Required Reading

  • UNIX Shell Basics: Learn the usage of head, tail, grep, awk, and pipes (|).

  • SSH Basics: Understand how to securely copy files between your local machine and a server using scp or rsync.

Johnson’s Rule: A 100-gigabyte genomic dataset does not fit into an Excel sheet, but it fits perfectly into a UNIX pipe. Master stream processing, or you will spend your career waiting for frozen laptops to reboot.
